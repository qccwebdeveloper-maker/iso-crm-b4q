1. System Overview

The ISO CRM is a full-stack web application built for Quality Control Certification (QCC) to manage the end-to-end ISO certification lifecycle — from lead capture to certificate issuance.

The system has five distinct user roles:

RoleResponsibilitiesPages
adminFull system control — manage users, applications, certificates, payments, reportsDashboard, Applications, Users, Auditors, Payments, Certificates, DMS, Leads, Standards, Roles, Observations, Audit Forms, Reports
clientSubmit applications, track progress, view documents and certificatesDashboard, Applications, Documents, Certificates, Feedback, Team Reports
auditorConduct audits, view assigned applications, submit audit reportsDashboard, Applications, Application Detail
reviewerReview completed audits (treated as auditor in the frontend, redirected to /auditor)Same as Auditor
salesManage leads, assign to team, convert leads to applicationsDashboard, Leads, Team, Assign, Reports, Applications List

The application status lifecycle follows this sequence:

draft → submitted → under_review → audit_stage1 → audit_stage2 → approved → certified
At any stage the admin can also move to rejected

6. Middleware Design

6.1 auth.js — protect()

Verifies the JWT token from the Authorization header. Attaches the decoded user object to req.user.

  • Reads token from Authorization: Bearer <token> header
  • Verifies signature using JWT_SECRET from environment
  • Fetches the full user from MongoDB by ID in the token payload
  • Returns 401 if token is missing, expired, or invalid
  • Returns 401 if user account is inactive

6.2 roleCheck.js — allow(...roles)

Takes an array of allowed roles. Returns a 403 if the logged-in user's role is not in the allowed list.

Usage example: allow('admin') or allow('admin', 'sales')

6.3 upload.js — Multer + Cloudinary

Configured with memory storage (no disk writes). Passes file buffer directly to Cloudinary via the cloudinary utility.

  • Accepts: PDF, DOC, DOCX, JPG, PNG
  • Maximum file size: 10 MB
  • Rejects invalid file types with 400 error before the route handler runs
  • Exposes req.file with buffer for the route to pass to Cloudinary

7. Cloudinary Integration

Cloudinary is used as the storage backend for all documents, PDFs, and images. Files are never stored on the server disk.

Cloudinary Folder Organization:

FolderContents
iso-crm/documents/All application documents (forms, agreements, reports)
iso-crm/certificates/Issued certificate files
iso-crm/avatars/User profile images

Upload Flow:

  1. Frontend sends multipart form with file + docType field
  2. Multer middleware parses the request, puts file in req.file.buffer
  3. Route handler calls uploadToCloudinary(buffer, folder, filename)
  4. Cloudinary returns secure_url and public_id
  5. Route saves both URL and public_id to MongoDB
  6. Named document fields (applicationForm, auditReport etc.) store just the URL
  7. A record is also inserted into the documents collection for the DMS view

Delete Flow:

  1. Admin deletes a document via DELETE /api/documents/:id
  2. Backend fetches the document record to get its Cloudinary public_id
  3. Calls Cloudinary's destroy API to remove the file from CDN
  4. Deletes the document record from MongoDB
Always store public_id alongside the URL. Without it, you cannot delete files from Cloudinary.

11. Key Design Decisions

DecisionChoice MadeReason
Notifications storageEmbedded array inside User documentNotifications are always loaded with the user. Array stays small — capped at 50 items per user.
Feedback storageEmbedded array inside Application documentFeedback is always read together with the application. Count stays small per app so no join needed.
Document storageCloudinary for files, MongoDB for metadataNo binary blobs in MongoDB. Cloudinary CDN handles delivery. public_id stored for deletion.
Dual document recordNamed URL fields on application + documents collectionNamed fields (applicationForm, auditReport etc.) for quick access. documents collection gives admin the DMS flat view without joins.
Denormalized namesraisedByName, uploadedByName stored as stringsAvoids population queries on read-heavy list views. Slight data duplication is acceptable.
Role-based data filteringApplied inside route handlers by checking req.user.roleAdmin sees all, client sees own, auditor sees assigned. Single endpoint — no separate routes per role.
Certificate settingsSingle document in certificateSettings collection with upsertThere is only ever one settings document. upsert on PUT avoids "not found" errors on first save.
applicationId formatAPP + padded 4-digit counter e.g. APP1004Human readable, sortable, used throughout the system including observations display.
Lead to Application conversionPOST /api/leads/:id/convertCreates Application, sets lead.convertedToApplication, sets lead.status to converted — all in one transaction.
File upload rejectionMulter validates type and size before route handlerInvalid files are rejected before any Cloudinary API call is made — saves cost and time.
